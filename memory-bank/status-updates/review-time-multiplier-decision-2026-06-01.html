Design checkpoint · Jun 1

Make the multiplier mean time, not free XP.

Current behavior scales only the XP number after one review action. The better long-term model is an actions-per-review tick: at 2x, a successful card runs two action attempts; at 3x, three attempts. Source XP stays honest, item chains move forward, success odds remain meaningful, and Utility/Activities benefit too.

Recommendation

Replace XP multiplier semantics with review time multiplier.

Keep `ankiscape_xp_multiplier` as a migrated config key if needed, but stop thinking of it as reward inflation. Runtime should ask, "how many action ticks does this review buy?" and then execute the same action pipeline repeatedly inside one undo-aware review transaction.

Backend Contract

1

One review, one transaction, N attempts.

Capture the pre-review `player_data` once, run up to N action ticks, then record one undo snapshot.

2

Stop on hard blockers.

Processing skills should stop when materials run out. Gathering skills can spend failed rolls normally.

3

Show aggregate feedback.

Floating XP should show the total earned across attempts, not one popup per sub-action.

Frontend Contract

1

Rename the setting.

Use "Review time multiplier" or "Actions per review"; avoid "XP multiplier" because it now lies.

2

Make it integer-first.

2x should be two actions. Fractional values can be deferred until there is a real need.

3

Explain material pressure.

Help text should say higher values spend resources faster and roll more gather attempts per card.

Balance Guardrails

Cap the UI tightly at first, likely 1x-5x or 1x-10x, because this setting accelerates the entire economy. Utility actions are already batched, so a time multiplier can compound with batch size. That is acceptable if it is named honestly and tested as elapsed review time.

Gathering: N independent rolls Processing: N recipes or until blocked Utility: N activity ticks, each respecting batch caps
