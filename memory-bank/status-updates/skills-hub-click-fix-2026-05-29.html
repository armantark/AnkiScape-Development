Skills Hub: clicks now move the panel

Clicking Woodcutting or Crafting left the target panel stuck on Mining. Root cause was a closure name collision, not the Qt widgets. Found and fixed with a new offscreen test — no Anki restart needed.

Root Cause — one variable, two definitions

The shadow

Both the hub and the Stats tab defined _select_skill as locals of the same show_main_menu function. Python keeps one binding per name per scope, so the later Stats-tab definition silently overwrote the hub's. Skill clicks were routed into the Stats selector, which never touches the hub panel.

ui.py:1177 — hub ui.py:1326 — stats

The fix

# before — shadowed by stats tab
def _select_skill(name):
  state["skill"] = name
  _render_panel()

# after — unique name
def _select_hub_skill(name):
  state["skill"] = name
  _render_panel()
The trace that cracked it
12:30:12hub.select_category: 'gathering'
12:30:12hub.render_panel: skill='Mining'
12:30:12hub.skill_row: name='Woodcutting' guard=False state_skill='Mining'  ← handler fired correctly
—hub.select_skill: 'Woodcutting'  ← NEVER LOGGED (call went to the wrong function)
—hub.render_panel: skill='Woodcutting'  ← so the panel stayed on Mining

The handler clearly fired with the right name and a true condition, yet no select_skill line followed. That gap pointed straight at a misrouted call rather than a Qt/signal problem.
